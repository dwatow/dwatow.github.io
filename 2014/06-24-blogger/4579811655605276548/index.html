試吃！這麼難？！

¶試吃！這麼難？！

原文連結: undefined
移植時的最後更新日期: 2014-06-24T16:25:00.440+08:00

菜鳥一進公司，還沒有寫程式，就有前輩在聊天時不經心的說出
「小菜，你過去，只有一個人自己寫，所以愛怎麼寫，就怎麼寫」

等小菜接第一個任務，小菜問了另一位前輩問題，前輩在看了小菜的程式(片段)時，發現小菜不使用公司準備好的marco而是將它們的程式碼另外抽取出來包成class之後，跟小菜說
「唉唉…如果有空，我看你還是改回主任工程師交待的 macro好了…¬_¬」
「不然到時候，不要說 code review會被念，我看連別人修改你的 code 的時候，也會被念了…」
「公司的程式會有多人協作的習慣，而且公司又很強調coding能在同一個原則下，所以，你的方式我也不能說你錯，只是…你能懂吧!?」
「『只要建立message的物件，再維護message、回傳狀態的對照表就做好了』不是這問題，是框架的問題，你現在程式框架和舊程式的框架已經不同了，如果今天，某人要在你的程式加功能的時候，勢必是要用你的框架的，如果是我要去加功能，我鐵定改回原舊程式的框架的…」
「而且這框架一開始是主任工程師定的…如果是他，也是會改回原框架」
「用try-catch層層轉拋，速度會比較慢，不如直接return」


過沒幾天，就請主管叫小菜改回來。

主管「你改了程式架構，改很多嗎？」
小菜說「很多，但是可以少寫很多程式碼，並且減少維護的範圍。」

主管問小菜「你下週一可以用舊架構進行整合測試嗎？」
小菜說「如果以舊架構，來不及。因為要多寫很多很多程式碼。」

小菜也著手開始改回來，才發現，真正讓前輩們抗拒的，並不是以下的原因。

  • 技術的好與不好。
  • 維護性。
  • 擴充性。
  • 可讀性。
  • 嚴慬程度。


而是要以下列原則，才不是個人風格：

  • 前輩過去的寫程式風格
  • 以及避免出現前輩看不懂的程式寫法


程式的優與劣，其實不是很喜歡用嘴巴講。
因為只要跑一次，只要請新手學習一次，就知道優劣了。

就像小當家講的「你吃一口就知道了」



講著講著，又餓了。
